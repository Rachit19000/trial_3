As you all know, the major problem today is 
  Manual creation of SDLC artifacts is slow, inconsistent, and error-prone which causes misalignment and rework.
  Teams spend significant time generating requirements, user stories, tech specs, test plans, and sprint plans.

To solve this we 
Need to convert raw requirements (PDF/DOCX/Text) into structured SDLC artifacts.
AI can generate artifacts but outputs must be human validated, versioned ( like if someone performs some change then all the corresponding documents must change ) , and integrated into workflows ( via the jira api ) 
Now our main task is to 
  Develop an AI-driven platform to automate the above functions
  Implement a workflow with human validation checkpoints.
  Use specialized AI agents for different artifact types.
  Integrate GitHub for version-controlled artifact storage.
Ensure quality via validation, retry logic, and structured outputs.

Scope
Frontend: React web interface for uploads, review, and approvals.
Backend: Java/Spring Boot API for orchestration and processing.
AI Agents: Python agents using LangGraph + HuggingFace models.
Integration: GitHub API for artifact versioning.
Communication: MCP (Model Context Protocol) between backend and AI agents.

Flow of the project
Frontend to backend (/ auth /github)
Frontend sends a request to the backend endpoint:
GET /auth/github
So the flow starts as:
User → Frontend → Backend (/auth/github)

Backend to Github ( redirect )
Backend generates state ( A random, unique string (e.g., UUID) generated by the backend ). This same state value acts as the CSRF protection token and is stored in the memory
Backend redirects user to GitHub OAuth ( official github login )
Backend now creates the GitHub authorization URL:
https://github.com/login/oauth/authorize
    ?client_id=GITHUB_CLIENT_ID
    &redirect_uri=http://localhost:3001/auth/github/callback
    &scope=repo,user
    &state=8f3a21c9-2d12-4b8b-9b8a-92c8a18b4c22
Backend responds with a redirect.
So the browser is redirected to GitHub.
Flow now becomes:
Frontend → Backend
Backend → Redirect → GitHub
User Browser → GitHub Login Page

CSRF (Cross-Site Request Forgery) means:
👉 A security attack where a malicious website tricks a logged-in user’s browser into sending unauthorized requests to another site.
How it works briefly:
The backend sends this random state to GitHub → GitHub sends it back in the callback → Backend checks if it matches the stored value → If it matches, the request is legitimate; if not, it’s rejected.
So, state = CSRF token, used to stop forged OAuth callbacks.

User authorizes on GitHub - GitHub redirects back to:
/api/v1/auth/github/callback?code=XXX&state=YYY

4. Backend handles OAuth callback
This is the most important step
a) Validate state (CSRF protection)
Checks if state exists in memory
Rejects if missing (possible attack)
Deletes it after use (one-time)
b) Exchange code → GitHub access token
Backend calls GitHub:
POST /login/oauth/access_token
Sends client_id, client_secret, code
Receives GitHub access token
c) Fetch GitHub user info
Calls:
GET https://api.github.com/user
Gets user data (id, username, email, avatar)
d) Create backend session token
Generates an internal session token
Maps it to:
GitHub access token
User details
Stores in TokenStore (in-memory)
✅ GitHub access token never leaves backend

5. Backend redirects to frontend
Redirects browser to:
frontend/oath/callback?token=SESSION_TOKEN&userData
👉 Only safe session token + user info is sent

6. Frontend stores session ( JWT token )
Extracts token from URL
Saves to localStorage
Uses token for future API calls:
Authorization: Bearer SESSION_TOKEN

Roles of the two tokens
🎫 Session Token (ghsession_...)
Used between Frontend ↔ Backend
Purpose:

Authorize API requests
Example:

🔑 GitHub Access Token
Used between Backend ↔ GitHub
Purpose:

Change Propagation Flow (Concise)
How it works
When you modify one document, change_propagation.py detects the change and calls the relevant agents to update related documents.
Propagation rules

Requirements changed  → Calls: user-stories-agent → tech-specs-agent
User Stories changed  → Calls: tech-specs-agent
Tech Specs changed    → Calls: user-stories-agent
Parsed doc changed    → Calls: user-req-agent → user-stories-agent → tech-specs-agent
Process
Detect change — file hash changes
Identify document type — _requirements.md, _user_stories.md, _tech_spec.md
Find related documents — matches by base name (e.g., Avni_SmartGovProj_Req_1)
Call agents — subprocess calls to regenerate dependent documents
Overwrite originals — saves to original file locations
Push to GitHub — commits updated files
Example

You edit: Avni_SmartGovProj_Req_1_requirements.md
         ↓
change_propagation.py detects change
         ↓
Calls user-stories-agent → Updates Avni_SmartGovProj_Req_1_user_stories.md
         ↓
Calls tech-specs-agent → Updates Avni_SmartGovProj_Req_1_tech_spec.md
         ↓
All files pushed to GitHub
Key methods
propagate_requirements_change() — requirements → stories → specs
propagate_user_stories_change() — stories → specs
propagate_tech_spec_change() — specs → stories
propagate_parsed_change() — parsed → requirements → stories → specs
Watch mode
Run python change_propagation.py watch to automatically detect and propagate changes as you edit files.Result: modifying one document updates all related documents via agent calls.
